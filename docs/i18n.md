# Internationalization (i18n)

How softwhere.uz serves three languages (Uzbek default, Russian, English) with
[next-intl](https://next-intl.dev) on the Next.js App Router: locale routing, message
bundles, translation helpers, locale switching, and how blog posts link across locales.

## At a glance

| Concern | Where | Value / notes |
| --- | --- | --- |
| Library | `next.config.mjs` | `createNextIntlPlugin('./src/core/i18n.ts')` wraps the config |
| Locales | everywhere (see below) | `en`, `ru`, `uz` |
| Default locale | `src/proxy.ts`, `src/core/i18n.ts`, `src/core/constants.ts` | `uz` |
| Locale prefix | routing | Always present in the URL: `/uz/...`, `/ru/...`, `/en/...` |
| Request config | `src/core/i18n.ts` | `getRequestConfig` — picks locale, dynamic-imports messages |
| Middleware | `src/proxy.ts` | www-strip + `/`→`/uz` + delegate to next-intl middleware |
| Messages | `src/messages/{en,ru,uz}.json` | One JSON file per locale, identical key trees |
| Document owner | `src/app/[locale]/layout.tsx` | Owns `<html lang>`, fonts, `NextIntlClientProvider` |
| Root layout | `src/app/layout.tsx` | Passthrough (returns `children`); no `<html>`/`<body>` |
| Cross-locale blog link | `generation_group_id` column | Ties the uz/ru/en variants of one article together |

> next-intl is wired in **locale-prefix "always"** mode with **no `localePrefix` or
> `pathnames` config** and **no `next-intl/navigation` helpers** — the codebase builds
> localized URLs by hand as `` `/${locale}/...` ``.

## The locale list lives in several places

There is no single exported `locales` constant; the same tuple is repeated. When you add
or rename a locale you must touch **all** of these:

| File | Symbol |
| --- | --- |
| `src/core/i18n.ts` | `const locales = ['en', 'ru', 'uz']` |
| `src/proxy.ts` | `createMiddleware({ locales: ['en','ru','uz'], defaultLocale: 'uz' })` and the regex `/^\/(en\|ru\|uz)(\/\|$)/` |
| `src/app/[locale]/layout.tsx` | `generateStaticParams()` and `hasLocale(['en','ru','uz'])` |
| `src/core/auth.ts` | `VALID_LOCALES = ['en','ru','uz']` + `isValidLocale` / `validateLocale` |
| `src/core/constants.ts` | `BLOG_CONFIG.SUPPORTED_LOCALES` and `BLOG_CONFIG.DEFAULT_LOCALE = 'uz'` |

`src/core/auth.ts` is the reusable validator used by API routes:

```ts
const VALID_LOCALES = ['en', 'ru', 'uz'] as const;
export function isValidLocale(locale: string): locale is ValidLocale { … }
export function validateLocale(locale: string | null, fallback: ValidLocale = 'en'): ValidLocale { … }
```

## Request config: `src/core/i18n.ts`

The plugin loads this on every server render to resolve the active locale and its messages:

```ts
const locales = ['en', 'ru', 'uz'] as const;

export default getRequestConfig(async ({ requestLocale }) => {
  const requested = await requestLocale;
  const locale = hasLocale(locales, requested) ? requested : 'uz';
  return {
    locale,
    messages: (await import(`../messages/${locale}.json`)).default,
  };
});
```

- `requestLocale` comes from the `[locale]` route segment (set by the middleware / by
  `setRequestLocale`, below). An unknown value falls back to `uz`.
- Messages are dynamically imported per locale, so only the active bundle ships.

## Routing: `[locale]` segment and layouts

Every user-facing page lives under `src/app/[locale]/…` (`page.tsx`, `blog/`, `estimator/`,
`admin/`, `feed.xml/`). The two layouts split responsibilities deliberately:

**`src/app/layout.tsx` — passthrough.** It sets `metadataBase` + default icons and imports
`globals.css`, then just returns `children`. It renders **no** `<html>`/`<body>` so the app
never nests two documents:

```ts
export default function RootLayout({ children }: { children: React.ReactNode }) {
  return children; // the [locale] layout is the sole document owner
}
```

**`src/app/[locale]/layout.tsx` — the real document.** It:

1. `generateStaticParams()` returns `[{locale:'uz'},{locale:'ru'},{locale:'en'}]` so the
   three shells are statically generated.
2. Validates the param and 404s bad locales at runtime:
   ```ts
   if (!hasLocale(['en', 'ru', 'uz'] as const, locale)) notFound();
   ```
   The header comment warns **not** to add `dynamicParams = false` here — on a parent
   layout it would force every nested blog slug to be pre-generated and break on-demand ISR.
3. Calls `setRequestLocale(locale)` to keep the tree statically renderable (without it,
   next-intl reads `headers()` and forces dynamic rendering). **Every page under `[locale]`
   that renders statically repeats this call** — see `src/app/[locale]/page.tsx`,
   `blog/page.tsx`, `blog/[slug]/page.tsx`.
4. Renders `<html lang={locale}>` with the font CSS variables, then — inside `ThemeProvider` —
   wraps the tree in `NextIntlClientProvider messages={messages}` (from `getMessages()`), which
   in turn wraps `BlogProvider` (holding `Header`, `children`, and `Footer`).

Because the root layout renders no document, the global `src/app/not-found.tsx` supplies its
own minimal `<html lang='en'>`.

## Message bundles

`src/messages/en.json`, `ru.json`, `uz.json` share an identical nested key tree. Top-level
namespaces: `header`, `estimatorCTA`, `estimator`, `hero`, `services`, `discuss`, `projects`,
`contact`, `toastMessage`, `faq`, `trust`, `aiSpotlight`, `process`, `blog`, `footer`,
`metadata`.

Values are plain strings with **ICU placeholders** in `{braces}`, e.g.:

```json
"stepProgress": "Step {current} of {total}",
"weeksRange": "{min}–{max} weeks",
"viewProject": "View {name}"
```

Message keys **are** type-checked: `global.d.ts` augments next-intl's `AppConfig` with
`Messages: typeof uz` (and `Locale: 'en' | 'uz' | 'ru'`), so `t('…')` keys and namespaces are
validated at compile time against the Uzbek bundle. The catch: types only mirror `uz`, so a key
that exists in `uz.json` but is missing from `en.json`/`ru.json` is **not** caught by the compiler
— it still throws at runtime for the locale that lacks it.

### Add or change a key

1. Add the key to **all three** files (`en.json`, `ru.json`, `uz.json`) at the same path in
   the tree. A key present in one bundle but missing in another throws for the locale that
   lacks it.
2. Read it via the namespace, matching the top-level object:

```tsx
// Server component / page:
const t = await getTranslations('blog');          // or ({ locale, namespace })
t('relatedArticles');
t('categories'); // nested namespace also works: getTranslations('blog.categories')

// Client component:
const t = useTranslations('header');
t('services');
```

3. For placeholders, pass values as the second arg: `t('stepProgress', { current, total })`.

## Using translations

| Context | Import | Helper |
| --- | --- | --- |
| Server component / `generateMetadata` | `next-intl/server` | `getTranslations`, `getMessages`, `setRequestLocale` |
| Client component (`'use client'`) | `next-intl` | `useTranslations`, `useLocale` |

`generateMetadata` reads translations by explicit locale so it never depends on request
state, e.g. in `src/app/[locale]/layout.tsx`:

```ts
const t = await getTranslations({ locale, namespace: 'metadata' });
```

## Locale-aware links

The repo does **not** use next-intl's navigation helpers. Links are plain `next/link`
`Link`s with a hand-built, locale-prefixed `href`. `src/shared/components/Header/index.tsx`
reads the active locale from the route params and prefixes every href:

```tsx
const params = useParams();
const [lang, setLang] = useState<string>((params?.locale as string) || 'uz');
// …
<Link href={`/${lang}/blog`}>{t('blog')}</Link>
<Link href={`/${lang}#services`}>{t('services')}</Link>
```

Convention: **always prefix internal hrefs with the current locale** (`` `/${locale}/…` ``).
One exception in the Header — the brand logo links to bare `href='/'`, which the middleware
then 308-redirects to `/uz` (so it always lands on the Uzbek home, regardless of the current
locale).

## Locale switching (`changeLanguage`)

The `LanguageSwitcher` in the Header calls `changeLanguage(locale)`
(`src/shared/components/Header/index.tsx`). Two paths:

- **On a blog post** — the current post is published to `BlogContext`
  (`src/modules/blog/context/BlogContext.tsx`) by `BlogPostClient`
  (`src/modules/blog/components/BlogPostClient.tsx`) on mount. If `currentPost.generationGroupId`
  is set, `changeLanguage` calls `getRelatedPost(groupId, locale)`
  (`src/modules/blog/api/posts.ts` → `GET /api/blog/posts/related`) and, on success, routes to
  the sibling slug `` `/${locale}/blog/${slug}` ``. If there is no sibling in the target
  locale (or the call fails), it falls back to `` `/${locale}/blog` ``.
- **Anywhere else** — it swaps the first path segment for the new locale, keeping the rest:
  ```ts
  const pathSegments = window.location.pathname.split('/').filter(Boolean);
  const routeWithoutLocale = pathSegments.slice(1).join('/');
  router.push(`/${locale}/${routeWithoutLocale}`);
  ```
  This reads `window.location.pathname` only, so any `#hash` or query string on the current
  URL is dropped on switch.

## Middleware / proxy: `src/proxy.ts`

`proxy` runs before next-intl's own middleware and short-circuits two redirects into a single
308 before delegating:

```ts
const intlMiddleware = createMiddleware({ locales: ['en','ru','uz'], defaultLocale: 'uz' });

export default function proxy(req: NextRequest): NextResponse {
  const host = req.headers.get('host') || '';
  const { pathname } = req.nextUrl;
  const isWww = host.startsWith('www.');
  const isRoot = pathname === '/';

  if (isWww || isRoot) {                      // ONE 308: strip www AND/OR /→/uz
    const newUrl = new URL(req.url);
    if (isWww) newUrl.host = host.replace(/^www\./, '');
    if (isRoot) newUrl.pathname = '/uz';
    return NextResponse.redirect(newUrl, 308);
  }

  const isLocalePath = /^\/(en|ru|uz)(\/|$)/.test(pathname);
  if (!isLocalePath) return NextResponse.next(); // no locale prefix → pass through, no redirect

  return intlMiddleware(req);                  // already locale-prefixed → next-intl handles it
}

export const config = {
  matcher: ['/((?!api|_next|_vercel|robots\\.txt|sitemap\\.xml|favicon|icons|images|.*\\..*).*)'],
};
```

Behavior worth knowing:

- **`/` always redirects to `/uz`** (308), collapsed with www-stripping so
  `www.softwhere.uz/` redirects exactly once instead of twice.
- **No `Accept-Language` detection.** A request to a non-locale path (e.g. `/blog`) is *not*
  auto-prefixed — it falls through `NextResponse.next()` and 404s. Only `/` is redirected,
  and always to `uz`. The visitor's browser language does not choose the locale.
- `intlMiddleware` only runs for paths already prefixed with a valid locale.
- The `matcher` excludes `api`, `_next`, `_vercel`, `robots.txt`, `sitemap.xml`, `favicon`,
  `icons`, `images`, and anything with a file extension (`.*\..*`).

## Blog posts across locales: `generationGroupId`

The blog is a single Postgres table (`blog_posts`, `src/modules/blog/model/BlogPost.ts`).
The three language variants of one article are separate rows sharing a
`generation_group_id`:

```ts
generationGroupId: text('generation_group_id'),           // nullable
index('blog_posts_group_idx').on(table.generationGroupId), // sibling lookups
uniqueIndex('blog_posts_locale_slug_uq').on(table.locale, table.slug),
```

- **Assigned at generation** (`src/app/api/blog/generate/route.ts`): a fresh run mints
  `generationGroupId = uuidv4()`; a continuation reuses the incoming `body.generationGroupId`
  so later locales join the existing group.
- **Repository queries** (`src/modules/blog/model/posts.repository.ts`) — routes/pages must go
  through these, never Drizzle directly:
  | Function | Purpose |
  | --- | --- |
  | `getPublishedGroupSibling(groupId, locale)` | The published variant in a target locale (used by language switch) |
  | `getGroupSiblings(groupId)` | All published variants (used to build hreflang alternates) |
  | `listGroupLocales(groupId)` | Locales present in the group in any status (CLI resume) |
  | `getByGroupId(groupId)` | Any one row in the group (validate a continuation) |
- **Cross-locale API**: `GET /api/blog/posts/related?generationGroupId=…&locale=…`
  (`src/app/api/blog/posts/related/route.ts`) validates the locale with `isValidLocale`,
  caps the id at `MAX_GROUP_ID_LENGTH = 128`, and returns the sibling slug or 404.

### hreflang, canonicals, and locale-mismatch redirects

The blog post page (`src/app/[locale]/blog/[slug]/page.tsx`) uses the group to emit SEO
`languages` alternates and to keep URLs canonical:

- `generateMetadata` builds `alternates.languages` from `getGroupSiblings`, resolving each
  sibling to its canonical slug, and sets `x-default` to the default-locale
  (`BLOG_CONFIG.DEFAULT_LOCALE` = `uz`) variant.
- The page issues `permanentRedirect` when the requested `[locale]` doesn't match the post's
  actual `locale`, or when the slug isn't the canonical one for that locale.
- `src/app/sitemap.ts` groups posts by `generationGroupId` to attach hreflang alternates per
  URL, and `src/app/[locale]/feed.xml/route.ts` emits one RSS feed per locale.

Home and blog-list pages don't have per-post siblings, so they hard-code the alternates
directly (`x-default` → `uz`, plus `uz`/`ru`/`en`) in their `generateMetadata`.

### date-fns localization

Dates are formatted with per-locale `date-fns` locales (`ru`, `uz`; `en` uses the default),
and the format string itself is locale-aware — see `DATE_LOCALES` in
`src/app/[locale]/blog/[slug]/page.tsx` and `src/modules/blog/components/BlogListClient.tsx`:

```ts
format(new Date(post.createdAt), locale === 'en' ? 'MMMM dd, yyyy' : 'd MMMM yyyy',
        { locale: DATE_LOCALES[locale] });
```

## Gotchas

- Adding a locale means editing every file in the [locale-list table](#the-locale-list-lives-in-several-places),
  adding a `src/messages/<locale>.json`, and extending `generateStaticParams`.
- Missing a key in one bundle throws only for that locale — keep the three trees in sync.
- Message keys are type-checked against the `uz` bundle (`global.d.ts`), but a key present in
  `uz.json` yet missing from `en.json`/`ru.json` isn't caught by types — it throws at runtime for
  that locale.
- Forgetting `setRequestLocale(locale)` in a new `[locale]` page silently opts the route into
  dynamic rendering.
- Non-locale paths are **not** auto-prefixed by the middleware; link with an explicit locale
  or you'll 404.
